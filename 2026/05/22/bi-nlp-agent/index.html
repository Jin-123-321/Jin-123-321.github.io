自然语言驱动的 BI 分析系统

发表于2026-05-22|更新于2026-05-22|项目展示
|浏览量:

项目名片

维度 内容
业务问题 让非技术用户用中文自然语言查询销售数据库，自动生成分析图表和洞察
技术方案 LLM 智能路由（本地 Ollama + 云端 DeepSeek），自动生成并验证 SQL
核心能力 复杂查询（窗口函数/CTE/分组极值）、SQL 自动纠错、业务洞察生成
技术栈 React 18 · Node.js 22 · SQLite · Ollama · DeepSeek API
项目链接 GitHub Repository

核心成果

这是一个零 npm 依赖的全栈 BI 系统：前端 React 18 通过 CDN 加载，后端仅使用 Node.js 内置模块，数据库采用 SQLite 文件存储。最轻量的同时实现了复杂分析查询的自动生成。

系统最核心的突破在于从规则引擎到 LLM 驱动的架构升级。旧版使用 String.includes() 关键词匹配理解用户意图，无法处理”每个国家分别在什么月份销量最低”这类需要分组极值的复杂查询（返回 116 行全量排列而非 19 行分组结果）。升级后的 LLM 引擎能理解”每个……分别”的语义含义，自动生成带 PARTITION BY 的窗口函数 SQL。

智能路由是另一大亮点：简单查询（排名/汇总/趋势）路由到本地 Ollama（10s 响应，数据不出本机），复杂查询（窗口函数/CTE/同比环比）路由到 DeepSeek 云端（10-20s）。还有降级策略确保服务可用性。

技术架构

分层技术选型

层级 技术 选择理由
前端 React 18 + Chart.js CDN 加载，零构建步骤
后端 Node.js 22 + node:sqlite 仅使用内置模块，无依赖
LLM 本地 Ollama + qwen2.5:7b 简单查询，隐私优先
LLM 云端 DeepSeek-V3 API 复杂查询，能力更强
数据库 SQLite 2,747 行经典汽车销售数据

核心功能

自然语言 → SQL

系统能理解的查询类型与对应的 SQL 生成逻辑：

用户输入 生成的 SQL 关键逻辑
各产品线销售额 GROUP BY PRODUCTLINE 聚合
每个国家分别在什么月份销量最低 ROW_NUMBER() OVER (PARTITION BY COUNTRY) 窗口函数
2019年比2018年增长多少 LAG(v) OVER (ORDER BY YEAR) 同比
哪些月份销售波动最大 MAX(v) - MIN(v) as range_val GROUP BY MONTH

SQL 自动验证与纠错

LLM 生成的 SQL 可能包含错误，系统内置了自动验证机制：

  1. 生成 SQL → 在数据库上试执行
  2. 执行失败 → 将错误信息反馈给 LLM，要求修正
  3. 最多重试 2 次，确保可用性

业务洞察生成

不满足于展示数据，系统还会自动生成分析洞察：

  • 📈 趋势解读：环比变化率、最大增幅/降幅
  • 🏆 排名分析：领先者与差距分析
  • 💡 行动建议：基于数据的营销/运营建议

交互式可视化

  • 柱状图、折线图、饼图等，字段自由映射
  • 数据钻取：点击柱子深入下级维度
  • SQL 溯源面板：透明展示查询逻辑

架构升级：从规则引擎到 LLM

为什么必须升级？

典型失败：用户输入”各个国家分别在什么月份销量最低”，系统返回 116 行全量排列而不是 19 行分组极值。

升级后的处理方式

LLM 能理解”每个国家分别“意味着需要分组极值，生成正确的窗口函数 SQL。

本地运行

零依赖，只需 Node.js 22+：

技术不应是数据分析的门槛。自然语言 + LLM 让每个人都能用数据回答业务问题。
